# 实验 7: 动态分区压测

> **测试日期**: 2026-06-07
> **环境**: 5 节点 YugabyteDB RF=3（Docker Compose + tc netem 30/60/90/120/150ms）

---

## 1. 实验设计

### 1.1 目的

在持续写入过程中注入网络分区（隔离 region2），观测读写行为的三阶段变化。

### 1.2 测试流程

| 时间 | 事件 |
|------|------|
| Phase 1 (0-10s) | 正常运行：后台持续写入 + 前端定时读取 |
| Phase 2 (10-25s) | 隔离 region2 (yb-2)：iptables 双向 DROP |
| Phase 3 (25-30s) | 恢复 region2：iptables 规则清除 |

---

## 2. 结果

### 2.1 Phase 1: 正常运行

| 读 # | 延迟 | 写入状态 |
|------|------|---------|
| 1 | 233ms | ✅ 成功 |
| 2 | 342ms | ✅ 成功 |
| 3-10 | 279-312ms | ✅ 成功 |

**读 avg ≈ 284ms, 写入全部成功**

### 2.2 Phase 2: 隔离 region2

| 读 # | 延迟 | 写入状态 |
|------|------|---------|
| 大部分读取 | 约 230-303ms | ✅ 后台写入大部分成功 |
| 长尾读取 | 约 11s | ⚠️ leader/路由切换导致长尾 |

**新脚本从健康节点 yb-1 发起读写。隔离 region2 后，大部分操作继续成功，但可能出现短暂写失败和秒级长尾。**

### 2.3 Phase 3: 恢复 region2

- 写入立即恢复（网络连通后 INSERT 成功）
- 读取延迟恢复正常

---

## 3. 分析

### 3.1 长尾读取的原因

隔离 region2 后，部分 SELECT 会出现秒级长尾，原因包括：

1. tablet leader 或请求路径经过被隔离的 region2
2. 客户端等待旧路由/leader 超时后重试
3. iptables DROP 不发送 TCP RST，会放大单次请求的尾部延迟

新脚本客户端连接 yb-1，因此不会在整个隔离窗口内完全不可用。

### 3.2 读写可用性取决于 leader 和客户端路径

从健康节点 yb-1 发起请求时，RF=3 集群仍可形成多数派。大部分读写可以继续成功；若 tablet leader 或旧路由涉及 region2，可能出现短暂失败或长尾。

### 3.4 恢复后立即生效

iptables 规则清除后：
- 网络立即恢复（无需等待 Raft 重新选举）
- 写入和读取立即成功
- **RPO = 0**：已提交的写入不会丢失

---

## 4. 结论

| 验证项 | 结果 |
|--------|------|
| Phase 1 正常读取 | ✅ avg ~284ms |
| Phase 2 读取 | ✅ 大部分成功，可能出现秒级长尾 |
| Phase 2 写入 | ✅ 大部分成功，可能短暂失败 |
| Phase 3 恢复 | ✅ 立即恢复, RPO=0 |

**核心发现**: 从健康节点发起请求时，隔离单个 region 不会让 RF=3 集群整体不可用，但 leader/路由切换会造成短暂失败和显著长尾；恢复后集群重新收敛。
