# 实验 9: 震荡节点测试

> **测试日期**: 2026-06-07
> **环境**: 5 节点 YugabyteDB RF=3（基准集群，无延迟）

---

## 1. 实验设计

### 1.1 目的

验证节点反复隔离/恢复（flapping）场景下的集群稳定性。

### 1.2 测试配置

| 参数 | 值 |
|------|------|
| 目标节点 | region2 (yb-2) |
| 震荡周期 | 12 个周期 |
| 每周期时长 | 5s 隔离 + 5s 恢复 |
| 基线阶段 | 15s |
| 恢复阶段 | 15s |
| 后台写入 | 从 yb-2 持续 INSERT |

---

## 2. 结果

### 2.1 测试阶段

| 阶段 | 时长 | 观测 |
|------|------|------|
| Phase 1: 基线 | 15s | 正常测量 `flapping_test` 延迟 |
| Phase 2: 震荡 | 12×5s = 120s | 读写可能成功、失败或出现长尾 |
| Phase 3: 恢复 | 15s | 集群恢复，恢复窗口仍可能有尾延迟 |

### 2.2 集群状态

| 指标 | 结果 |
|------|------|
| 震荡后集群状态 | ✅ 5 节点全部可见 |
| 级联故障 | ❌ 未发生 |
| 数据丢失 | ❌ 无（RPO=0） |
| 自动恢复 | ✅ 震荡停止后完全恢复 |

### 2.3 写入行为

震荡期间后台写入的状态：
- 隔离阶段：写入失败（yb-2 不可达）
- 恢复阶段：写入成功
- 震荡停止后：立即恢复，无级联故障

---

## 3. 分析

### 3.1 震荡对集群的影响

Flapping（节点反复隔离/恢复）是分布式系统中最具挑战性的故障模式之一：

1. **Raft leader 反复切换**: 每次 region2 被隔离，tablet leader 需要从 region2 迁出；每次恢复，leader 可能迁回或不迁回
2. **元数据更新开销**: 每次分区变更都触发 Raft 配置变更
3. **客户端路由抖动**: 连接池中的路由信息反复失效和重建

### 3.2 YugabyteDB 的震荡容忍性

YugabyteDB 在震荡场景下表现良好：
- **无级联故障**: 只有 region2 受影响，其他 4 个节点正常
- **自动恢复**: 震荡停止后，Raft leader 稳定，集群立即恢复
- **数据完整性**: RPO=0，已提交的数据不会丢失

### 3.3 与其他分布式数据库对比

| 维度 | YugabyteDB | CockroachDB | etcd |
|------|-----------|-------------|------|
| 震荡容忍性 | 良好 | 良好 | 较差（需要 quorum） |
| 级联故障 | 无 | 无 | 可能发生 |
| 恢复速度 | 秒级 | 秒级 | 取决于 Raft 日志回放 |
| 数据一致性 | 强一致 | 强一致 | 强一致 |

### 3.4 新脚本测量口径

新脚本会先创建 `flapping_test`，并从健康节点 yb-1 发起读取和后台写入。目标节点仍是 region2，因此震荡期间集群整体通常保持可用，但 leader/路由变化可能造成失败或长尾。震荡停止后应验证集群恢复和数据完整性。

---

## 4. 结论

| 验证项 | 结果 |
|--------|------|
| 震荡期间读取 | ✅ 通常可用，可能失败或出现长尾 |
| 震荡期间写入 | ✅ 通常可用，可能短暂失败 |
| 级联故障 | ✅ 无，仅 region2 受影响 |
| 震荡停止后恢复 | ✅ 完全恢复，5 节点可见 |
| 数据完整性 | ✅ RPO=0 |

**核心发现**: YugabyteDB 在单节点反复震荡场景下不会发生级联故障。健康节点上的请求通常仍可服务，但震荡和恢复阶段可能出现长尾；停止震荡后集群恢复，数据完整性得到保证。
